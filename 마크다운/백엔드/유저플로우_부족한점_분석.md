# 유저 플로우 부족한 점 분석

이 문서는 이미지 플로우차트와 현재 유저 플로우 문서를 비교하여 부족한 점을 정리합니다.

---

## 1. 게임 시작 단계의 상세 설명 부족

### 현재 상태
- "게임 시작"만 표시되어 있음
- 게임 초기화 과정이 명확하지 않음

### 부족한 점
- **시나리오 선택**: 사용자가 어떤 시나리오를 선택하는지
- **게임 생성 API 호출**: `GET /api/v1/scenario/start/{scenario_id}?user_id={user_id}` 호출 과정
- **게임 상태 초기화**: 초기 턴 수, 인간성, NPC 상태 등 초기값 설정
- **첫 씬 전환**: Title 씬 → PlayersRoom 씬 전환 과정

### 권장 추가 내용
```
[게임 시작]
    │
    ▼
[시나리오 선택 화면]
    │
    ▼
[게임 생성 API 호출]
GET /api/v1/scenario/start/{scenario_id}?user_id={user_id}
    │
    ▼
[게임 상태 초기화]
- game_id 설정
- 턴 수 초기화 (예: 10턴)
- 인간성 초기화 (예: 100)
- NPC 상태 초기화
- 아이템 상태 초기화
    │
    ▼
[PlayersRoom 씬으로 전환]
```

## 2. 턴 관리 메커니즘의 상세 설명 부족

### 현재 상태
- "턴 소모 (remainingTurns 감소)"만 언급
- 턴 관리의 세부 사항이 불명확

### 부족한 점
- **턴 소모 시점**: 언제 턴이 소모되는지 (API 호출 시? 응답 받은 후?)
- **턴 소모 조건**: 어떤 행동이 턴을 소모하는지 (장소 이동은 턴 소모 안 함)
- **턴 소진 처리**: 턴이 0이 되었을 때의 정확한 처리 순서
- **턴 리셋 조건**: 언제 턴이 리셋되는지 (다음 날 시작 시)

### 권장 추가 내용
```
턴 관리:
- 초기 턴 수: 10턴
- 턴 소모 조건:
  * NPC와 상호작용: 1턴 소모
  * 아이템 사용/습득: 1턴 소모
  * 장소 이동: 턴 소모 없음
- 턴 소진 시:
  1. 턴이 0이 되면 자동으로 Night 씬으로 전환
  2. Night 씬에서 POST /night_dialogue 호출
  3. 대화 완료 후 다음 날로 진행
  4. 턴 수 리셋 (10턴으로 복구)
```

## 3. 장소 이동 메커니즘 설명 부족

### 현재 상태
- "장소 이동"이 행동 리스트에만 언급됨
- 장소 이동의 세부 메커니즘이 없음

### 부족한 점
- **장소 이동 방법**: 어떻게 장소를 이동하는지 (숫자키? UI 버튼?)
- **장소 이동 시 턴 소모 여부**: 턴을 소모하지 않는다는 명시 필요
- **장소 이동 시 상태 변화**: 위치 변경이 게임 상태에 미치는 영향
- **장소별 특성**: 각 장소에서 할 수 있는 행동의 차이

### 권장 추가 내용
```
장소 이동:
- 방법: 숫자키(1~7) 또는 UI 버튼
- 턴 소모: 없음
- 위치 추적: GameStateManager의 LocationManager가 관리
- 씬 전환: SceneFadeManager를 통한 페이드 효과
- 사용 가능한 장소:
  * 1: PlayersRoom (주인공 방)
  * 2: Hallway (복도)
  * 3: LivingRoom (거실)
  * 4: Kitchen (부엌)
  * 5: SiblingsRoom (동생 방)
  * 6: Basement (지하실)
  * 7: Backyard (마당)
```

## 4. NPC State의 무작위성 도입 설명 부족

### 현재 상태
- 이미지 플로우차트에 "NPC State의 무작위성 도입"이 명시되어 있음
- 현재 문서에는 이 내용이 없음

### 부족한 점
- **무작위성의 의미**: 밤 시간에 NPC 상태가 어떻게 무작위로 변하는지
- **무작위성 적용 범위**: 어떤 NPC 상태에 무작위성이 적용되는지
- **무작위성의 목적**: 왜 무작위성을 도입하는지 (게임 난이도 조절? 재플레이 가치?)

### 권장 추가 내용
```
밤 시간 - NPC State 무작위성:
- 목적: 낮 동안의 상호작용 기반 평가에 무작위 요소 추가
- 적용 범위:
  * NPC 호감도(trust) 변화에 ±5% 랜덤 변동
  * NPC 의심도(suspicion) 변화에 ±3% 랜덤 변동
  * NPC 공포도(fear) 변화에 ±2% 랜덤 변동
- 효과: 같은 행동을 해도 결과가 약간씩 달라져 재플레이 가치 증가
```

## 5. 다음 날 진행(AdvanceToNextDay) 메커니즘 상세 설명 부족

### 현재 상태
- "대화 완료 후 다음 날로 진행 (AdvanceToNextDay)"만 언급
- 상세 과정이 불명확

### 부족한 점
- **AdvanceToNextDay 호출 시점**: 정확히 언제 호출되는지
- **다음 날 진행 시 처리 사항**: 
  * 날짜 증가
  * 인간성 10% 감소
  * 턴 리셋
  * 아이템 리스폰
  * NPC 무력화 상태 업데이트
  * 플래그 처리 (예: tea_with_sleeping_pill)
- **게임 오버 체크**: 인간성이 0 이하로 떨어졌을 때의 처리
- **최대 일수**: 5일차 종료 시 처리

### 권장 추가 내용
```
다음 날 진행 (AdvanceToNextDay):
1. Night 대화 완료 후 자동 호출
2. 날짜 증가 (currentDay++)
3. 인간성 10% 감소 (ModifyHumanity(-10f))
4. 게임 오버 체크:
   - 인간성 ≤ 0 → UnfinishedDoll 엔딩 트리거
5. 최대 일수 체크:
   - currentDay > MAX_DAY (5일) → EternalDinner 엔딩 트리거
6. 다음 날 초기화:
   - 턴 수 리셋 (10턴으로 복구)
   - 시간대를 Day로 설정
   - 아이템 리스폰 (RespawnDailyItems)
   - 플래그 처리 (예: tea_with_sleeping_pill → 가족 3턴 무력화)
7. Tutorial 씬으로 전환 (페이드 효과)
```

## 6. 에러 처리 및 예외 상황 플로우 부족

### 현재 상태

### 부족한 점
- **API 호출 실패**: 네트워크 에러, 타임아웃 시 처리
- **잘못된 입력**: 사용자가 잘못된 입력을 했을 때의 처리
- **게임 상태 불일치**: 프론트엔드와 백엔드 간 상태 불일치 시 처리
- **폴백 엔딩**: 에러 발생 시 폴백 엔딩 트리거 조건

### 권장 추가 내용
```
에러 처리:
1. API 호출 실패:
   - 네트워크 에러: 목업 데이터 반환 또는 재시도
   - 타임아웃: 목업 데이터 반환

3. 폴백 엔딩:
```

- **상태 동기화**: 백엔드 응답의 state_result를 프론트엔드에 어떻게 적용하는지

### 권장 추가 내용
```
낮 시간 루프:
[사용자 입력] 
  → [POST /api/v1/game/{gameId}/step]
  → [백엔드 응답 수신]

밤 시간 루프:
[POST /api/v1/game/{gameId}/night_dialogue]
  → [백엔드 응답 수신]
  → [state_result 적용]
  → [dialogues[] 순차 표시]
  → [대화 완료]
  → [AdvanceToNextDay 호출]
  → [다음 날 초기화]
```

- **엔딩 처리 순서**: ending_info를 받았을 때의 처리 순서
- **엔딩 씬 전환**: 엔딩 씬으로 전환하는 시점과 조건

### 권장 추가 내용
```
엔딩 처리:
1. 엔딩 트리거:
```

## 9. 게임 상태 동기화 메커니즘 설명 부족

### 현재 상태
- 게임 상태 변화는 언급되지만, 프론트엔드와 백엔드 간 동기화 방법이 불명확

### 부족한 점
- **상태 동기화 주체**: 누가 상태를 동기화하는지 (프론트엔드? 백엔드?)
- **상태 동기화 시점**: 언제 동기화가 일어나는지
- **상태 충돌 해결**: 프론트엔드와 백엔드 상태가 다를 때의 처리

### 권장 추가 내용
```
게임 상태 동기화:
- 원칙: 백엔드가 단일 진실의 원천(Single Source of Truth)
- 동기화 시점: 백엔드 API 응답 수신 시
- 동기화 방법:
  1. 백엔드 응답의 state_result를 프론트엔드 GameStateManager에 적용
  2. 변경되지 않은 필드는 생략 가능 (백엔드가 최신 상태만 반환)
  3. 프론트엔드는 백엔드 응답을 신뢰하고 로컬 상태를 업데이트
```

## 11. NPC 무력화 상태 관리 상세 설명 부족

### 현재 상태
- npc_disabled_states가 언급되지만, 관리 메커니즘이 불명확

### 부족한 점
- **무력화 상태 설정**: 언제 NPC가 무력화되는지
- **무력화 상태 업데이트**: 턴이 지날 때마다 remaining_turns가 어떻게 감소하는지
- **무력화 해제**: remaining_turns가 0이 되었을 때의 처리
- **무력화 상태 표시**: UI에서 어떻게 표시되는지

### 권장 추가 내용
```
NPC 무력화 상태 관리:
- 설정: 백엔드 응답의 state_result.npc_disabled_states
- 구조:
  {
    "npc_name": {
      "is_disabled": true,
      "remaining_turns": 3,
      "reason": "수면제 복용"
    }
  }

- 업데이트:
  * 매 턴 소모 시 remaining_turns 감소
  * remaining_turns가 0이 되면 is_disabled = false로 변경
  * 무력화된 NPC는 씬에 표시되지 않음

- 해제:
  * remaining_turns = 0
  * 또는 백엔드 응답에서 is_disabled = false로 명시
```

### 권장 추가 내용
```
아이템 시스템:

  * InWorld: 장소에 있음
  * InInventory: 인벤토리에 있음
  * Used: 사용됨
```

## 종합 권장 사항

1. **게임 시작 단계 상세화**: 시나리오 선택부터 첫 씬 전환까지의 전체 과정
2. **턴 관리 메커니즘 명확화**: 턴 소모 조건, 소진 처리, 리셋 조건
3. **장소 이동 시스템 설명 추가**: 방법, 턴 소모 여부, 위치 추적
4. **NPC State 무작위성 설명 추가**: 이미지에 명시된 내용 반영
5. **AdvanceToNextDay 상세화**: 다음 날 진행 시 모든 처리 사항
6. **에러 처리 플로우 추가**: 예외 상황 처리 방법
7. **루프 구조 명확화**: 낮/밤 시간 루프의 정확한 흐름
8. **엔딩 처리 통합**: 메인 플로우에 엔딩 트리거 및 처리 과정 포함
9. **상태 동기화 메커니즘 설명**: 프론트엔드-백엔드 동기화 방법
10. **플래그 처리 상세화**: 플래그 설정, 처리, 해제 과정
11. **NPC 무력화 상태 관리 상세화**: 설정, 업데이트, 해제 과정
12. **아이템 시스템 설명 추가**: 획득, 사용, 리스폰, 상태 관리
